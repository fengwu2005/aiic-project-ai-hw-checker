# 设计说明：CoCode Viva

## 一、设计目标

CoCode Viva 的目标是帮助大学教师判断学生是否真正掌握编程作业，而不是替代教师评分或做简单查重。系统通过班级作业管理、固定材料格式、隐藏验收、有限轮次答辩、可解释评分和教师审核，判断学生对终版代码和报告的理解程度。

当前固定作业为 ImageLab，要求学生使用 Pillow/PIL 完成一个图像变换工具。必做功能包括图片读取、保存、放大、缩小、旋转、裁剪、反色、均值模糊、固定卷积核边缘提取和中值滤波；这些都是固定必做功能，不属于 bonus。

## 二、整体流程

1. 用户选择学生端或教师端，并注册/登录账号。
2. 学生使用班级邀请码加入班级，进入班级作业列表。
3. 学生阅读固定作业题目 ImageLab，并上传 zip 提交包。
4. 系统安全解压提交包。
5. 系统按固定文件名读取 `README.md`、`final/image_ops.py` 和 `report/report.md`。
6. 本地 skills 对材料进行结构化分析。
7. HiddenTestSkill 在临时目录执行教师隐藏验收，验证终版图像代码真实行为。
8. 系统只生成第 1 个短答辩问题。
9. 学生逐题回答，每次只看到当前问题；学生可以查看和下载自己的提交文件，但教师审核前不看到评分报告、隐藏验收明细或后台证据。
10. 每次回答后，API Agent 或本地规则结合材料证据、隐藏验收结果和历史回答决定下一问。
11. 系统输出评分报告与掌握度判断到教师端。
12. 教师查看学生提交文件、报告和后台证据，必要时下载提交 zip，审核并保存最终分数。
13. 教师审核后，学生端展示最终分数、教师评语和面向学习复盘的系统反馈；后台证据、风险排序和隐藏验收明细仍仅教师可见。

## 三、为什么固定作业和固定格式

- 答辩问题可以更有针对性，不会变成泛泛聊天。
- 文件命名固定后，本地工具能可靠读取学生材料。
- 隐藏验收能覆盖固定函数接口，评分标准更稳定。
- 教师端能按风险排序快速复核重点提交。

## 四、本地 skills 设计

### ArchiveSkill

负责安全解压 zip 文件。它会限制压缩包大小、单文件大小、文件数量，并检查路径穿越，避免恶意 zip 写出目标目录。

### FileReaderSkill

负责按固定命名读取：

- `README.md`
- `final/image_ops.py`
- `report/report.md`

### CodeAnalysisSkill

负责解析 Python 代码结构和功能关键词。当前实现会提取有效代码行数、函数、类、导入模块，以及是否包含图像读取保存、CLI、缩放、旋转、裁剪、反色、模糊、边缘检测、中值滤波、像素访问、参数校验和错误处理。

### HiddenTestSkill

负责在代码层面对学生提交的 `final/image_ops.py` 进行教师隐藏验收。它会在临时目录中导入学生代码，并调用作业要求中固定的函数接口。

隐藏验收覆盖图片读写、缩放、旋转、裁剪、反色、均值模糊、边缘检测、中值滤波、统一调度和参数校验。它给 Agent 和评分模块提供可复现的代码行为证据。

### InteractionSkill

负责分析报告质量。它关注报告是否说明实现方法、验证方式、个人实现说明、限制不足，以及是否出现“函数、验证、边界、像素、卷积、滤波”等能支撑答辩的问题证据。

### QuestionSkill

负责提供本地候选问题池。问题覆盖：

- 图像代码理解
- 实现方法
- 系统验收
- 边界情况
- 报告证据和掌握情况

每个问题尽量绑定可核查证据，例如实现位置、隐藏验收项、报告段落或运行结果；答辩评分首先看学生是否答中了当前问题、理解是否正确、解释是否自洽，证据定位只是辅助判断。

### AgentToolSkill

负责为系统助教 Agent 提供受控本地工具。配置 OpenAI-compatible API 后，Agent 不会直接访问任意文件，而是通过 JSON 输出请求调用以下工具：

- `list_materials`：列出可读取的提交材料
- `read_material`：按材料 key 和行号读取指定内容
- `search_material`：在指定材料中搜索关键词
- `get_static_analysis`：读取本地静态分析和隐藏验收摘要

### DefenseAgent

DefenseAgent 是系统助教调度层。当配置 API key 时，流程如下：

1. 系统先通过本地 skills 解压、读取、静态分析并执行隐藏验收。
2. Agent 获得材料索引和分析摘要。
3. Agent 输出结构化 `tool_requests`，说明需要调用哪些本地工具和原因。
4. 系统执行本地工具，返回代码片段、报告片段和隐藏验收结果。
5. Agent 基于这些证据生成个性化答辩问题。
6. 学生回答后，Agent 再次调用工具索取证据，并以 JSON 输出 `ask` 或 `end_defense` 决策。
7. 系统将 Agent 建议和本地规则评分合并，输出最终评分参考。

如果未配置 API key，系统自动降级为本地规则。

### ScoringSkill

负责生成评分参考。它综合：

- 提交文件是否完整
- 图像功能证据
- 教师隐藏验收结果
- 报告中的实现、验证和个人实现说明
- 答辩回答是否具体、是否能对应代码、报告与隐藏验收

## 五、答辩逻辑

系统采用“证据交叉验证”的答辩逻辑：

- 如果代码中出现某图像功能，但学生无法解释，说明理解风险较高。
- 如果学生说自己实现或验证了某问题，但隐藏验收、代码和报告中没有证据，说明掌握证据不足。
- 如果报告只写宏观反思，没有函数、输入、输出、边界或验收说明，说明报告证据偏弱。
- 如果学生能把回答落到 `final/image_ops.py` 的函数、报告段落和隐藏验收项，说明掌握证据更强。

现场答辩是验证学生本人掌握程度的硬门槛。每题只问一个小点，最多 6 轮；学生没理解当前问题时直接在回答框里追问题意，系统判断为澄清请求后改写当前问题，澄清不计入正式答题轮次。评分模块设置答辩有效性闸门：

- 答辩无效：多题空话、重复短句或连续表示不会；即使作业包完整，总分通常最高 35 到 42。
- 答辩较弱：能回答少量信息但缺少代码、图像概念和系统验收证据，总分最高 55。
- 答辩部分有效：能解释部分内容但证据链不完整，总分最高 72。
- 答辩有效：材料质量、隐藏验收结果和报告证据才充分转化为高分。

## 六、风险排序

教师端按风险优先展示提交，主要风险包括：

- 答辩无效或较弱
- 隐藏验收未全过
- 报告空泛
- 实现说明不足
- 验证说明不足
- 代码强但答辩弱

## 七、可解释性

学生端可查看和下载自己的提交文件；教师审核前只展示作业要求、上传答辩流程和提交状态；教师审核后展示最终分数、教师评语、系统助教反馈、优势与改进点、分项参考和逐题答辩反馈。后台证据、风险排序和隐藏验收明细仍仅教师可见。教师端可查看学生提交文件并下载提交 zip；教师端报告展示系统参考分、分项分、优势证据、风险提示、逐题答辩评分解释和教师审核表单。后台证据面板展示逐题记录、工具调用、隐藏验收明细和日志快照。

## 八、局限与改进

当前系统已经支持本地规则模式和 API Agent 模式，但仍不能完全替代教师判断。后续可改进方向：

- 在更强沙箱中运行隐藏验收并支持更多题型
- 对图像代码进行更细粒度静态分析
- 记录答辩耗时和回答修改轨迹
- 加入班级内相似度分析
- 使用大模型对答辩回答进行更精细的语义评分
