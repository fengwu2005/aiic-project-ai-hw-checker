# 设计说明：CoCode Viva

## 一、设计目标

CoCode Viva 的目标不是检测学生是否使用 AI，而是评价学生是否具备“负责任地使用 AI 完成编程任务”的能力。系统通过固定作业、固定材料格式、有限轮次答辩和可解释评分，判断学生的真实理解程度和个人贡献比例。当前固定作业为 TaskFlow Pro，要求学生完成一个包含组合筛选、搜索排序、批量操作、导入导出和统计能力的命令行任务管理器；这些都是固定必做功能，不属于 bonus。作业同时允许学生结合 AI 自主设计额外扩展功能，作为 AI 共创能力和原创性的 bonus 证据。

## 二、整体流程

1. 学生阅读固定作业题目 TaskFlow。
2. 学生上传 zip 提交包。
3. 系统安全解压提交包。
4. 系统按固定文件名读取最终代码、第一次 AI prompt、第一次 AI 回复、AI 初版代码、完整对话记录和学生报告。
5. 本地 skills 对材料进行结构化分析。
6. HiddenTestSkill 在临时目录执行教师隐藏验收，验证最终代码真实行为。
7. 系统只生成第 1 个答辩问题。
8. 学生逐题回答，每次只看到当前问题。
9. 每次回答后，API Agent 接收已问问题、全部回答、上一轮回答、材料证据和隐藏验收结果，结构化决定下一步。
10. Agent 可以输出继续追问、切换维度或提前终止答辩；本地规则只在 API 不可用时兜底。
11. 系统输出评分报告与原创性判断。

## 三、为什么固定作业和固定格式

本项目的重点是答辩逻辑，而不是通用 LMS 平台。固定作业有三个好处：

- 答辩问题可以更有针对性，不会变成泛泛聊天。
- 文件命名固定后，本地工具能可靠读取学生材料。
- 评分标准更稳定，系统输出更容易解释。

## 四、本地 skills 设计

系统将“读材料”拆成多个本地 skills，再把结构化摘要交给动态追问和评分模块。

### ArchiveSkill

负责安全解压 zip 文件。它会限制压缩包大小、单文件大小、文件数量，并检查路径穿越，避免恶意 zip 写出目标目录。

### FileReaderSkill

负责按固定命名读取：

- `README.md`
- `final/taskflow.py`
- `ai/initial_prompt.md`
- `ai/initial_response.md`
- `ai/initial_code.py`
- `ai/full_conversation.md`
- `report/report.md`

### CodeAnalysisSkill

负责解析 Python 代码结构和功能关键词。当前实现会提取：

- 有效代码行数
- 函数、类、导入模块
- 是否包含 JSON 持久化
- 是否包含命令行接口
- 是否处理优先级、截止日期、标签、关键词搜索、排序、统计、归档、批量操作、导入导出、状态筛选、id 生成、异常处理

### HiddenTestSkill

负责在代码层面对学生提交的 `final/taskflow.py` 进行教师隐藏验收。它会启动一个有超时限制的 Python 子进程，在临时目录中导入学生代码，并调用作业要求中固定的函数接口：

- `add_task`
- `list_tasks`
- `update_status`
- `complete_task`
- `delete_task`
- `batch_update_status`
- `archive_tasks`
- `export_tasks`
- `import_tasks`
- `task_statistics`

隐藏验收覆盖任务字段、JSON 持久化、非法输入异常、组合筛选、关键词搜索、排序、批量更新、归档、导入导出重复 id 处理和统计摘要。它不是替代 AI 助教，而是给 Agent 和评分模块提供可复现的代码行为证据。

### InteractionSkill

负责分析 AI 协作记录。它关注：

- 是否至少 5 轮有效交互
- 初始提示是否具体
- 是否出现“我发现、我决定、边界、修改、验证”等个人思考证据
- 学生报告是否提到 AI 局限、个人修改、实现方法和迭代记录

### QuestionSkill

负责提供本地候选问题池。动态追问会从候选池中选择下一问，或在回答证据不足时生成针对上一问的追问。问题覆盖：

- 代码理解
- AI 协作过程
- 系统验收
- 边界情况
- 原创性与个人贡献

### AgentToolSkill

负责为 AI 助教 Agent 提供受控本地工具。配置 OpenAI-compatible API 后，AI 不会直接访问任意文件，而是通过 JSON 输出请求调用以下工具：

- `list_materials`：列出可读取的提交材料
- `read_material`：按材料 key 和行号读取指定内容
- `search_material`：在指定材料中搜索关键词
- `compare_initial_final_code`：比较 AI 初版代码和学生最终代码
- `get_static_analysis`：读取本地静态分析和隐藏验收摘要

这个设计让 AI 像真正助教一样“先看证据，再提问和评分”。同时，工具是白名单式的，只能读取本次提交包中固定命名的材料，避免任意文件访问。

### DefenseAgent

DefenseAgent 是系统的 AI 助教调度层。当配置 API key 时，流程如下：

1. 系统先通过本地 skills 解压、读取、静态分析并执行隐藏验收。
2. AI Agent 获得材料索引和分析摘要。
3. AI Agent 输出结构化 `tool_requests`，说明需要调用哪些本地工具和原因。
4. 系统执行本地工具，返回代码片段、隐藏验收结果、对话片段或初版/终版 diff。
5. AI Agent 基于这些证据生成个性化答辩问题。
6. 学生回答后，AI Agent 再次调用工具索取证据，并以 JSON 输出 `ask` 或 `end_defense` 决策。
7. 若输出 `ask`，系统展示 Agent 给出的下一问；若输出 `end_defense`，系统提前结束答辩并记录原因。
8. 系统将 AI Agent 建议和本地规则评分合并，输出最终评分参考。

如果未配置 API key，系统自动降级为本地规则：仍可读取提交材料、生成答辩问题和输出报告，只是流程决策、问题深度和报告语义不如 Agent 模式。

### ScoringSkill

负责生成评分参考。它不会只看最终代码，而是综合：

- 提交文件是否完整
- 代码功能证据
- 教师隐藏验收结果
- 系统验收证据
- AI 协作过程证据
- 学生报告证据
- 答辩回答是否具体、是否能对应代码与系统验收

## 五、答辩逻辑

系统采用“证据交叉验证”的答辩逻辑：

- 如果代码中出现某功能，但学生无法解释，说明理解风险较高。
- 如果学生说自己修复了某问题，但隐藏验收、代码和报告中没有证据，说明贡献证据不足。
- 如果 AI 交互记录只有简单命令，没有目标、验证和取舍，说明 AI 协作质量偏低。
- 如果学生能指出 AI 初版的具体缺陷，并说明自己如何验证和修改，说明个人贡献较高。
- 如果学生自主提出扩展功能，并能提供 README、报告和答辩解释中的一致证据，可作为 AI 共创 bonus 和个人贡献的正向信号。

现场答辩不是简单加分项，而是验证学生本人掌握程度的硬门槛。API Agent 会结合历史回答决定是否继续追问或提前终止；评分模块再设置答辩有效性闸门：

- 答辩无效：例如多题回答只有 `1`、空话或重复短句，总分最高 45，个人贡献比例最高 35%。
- 答辩较弱：能回答少量信息但缺少代码和系统验收证据，总分最高 60，个人贡献比例最高 50%。
- 答辩部分有效：能解释部分内容但证据链不完整，总分最高 75，个人贡献比例最高 70%。
- 答辩有效：才允许材料质量、隐藏验收结果和 AI 协作记录充分转化为高分。

这样可以避免“作业包看起来很好，但学生现场完全答不上来仍然高分”的不合理情况。

Agent 模式下，AI 助教不会只接收一段摘要后直接评分，而是会按需索取证据。例如：

```json
{
  "tool_requests": [
    {
      "tool": "compare_initial_final_code",
      "args": {"max_lines": 180},
      "reason": "需要比较 AI 初版代码和最终代码，判断学生关键修改"
    },
    {
      "tool": "get_static_analysis",
      "args": {},
      "reason": "需要查看隐藏验收和静态分析结果"
    }
  ]
}
```

系统执行这些工具后，AI 才生成问题或评分参考。这样可以避免 AI 泛泛提问，也能让报告中的判断更可追溯。

## 六、原创性判定方法

系统不输出“抄袭/未抄袭”的绝对结论，而输出个人贡献比例估计。判断依据包括：

- 代码复杂度与答辩解释能力是否匹配
- AI 交互记录是否体现学生主动提出约束和反馈
- 学生报告是否具体到函数、系统验收要求、AI 初版问题和个人贡献
- 学生是否能解释为什么接受或拒绝 AI 建议
- 学生是否能现场解释边界行为或改进方案
- 最终代码、隐藏验收、报告和答辩回答之间是否一致

输出示例：

```text
个人贡献比例估计：72%，水平：中等偏高。
依据：学生能解释日期校验、验收隔离和异常处理修改，也能指出 AI 初版接口设计问题；
但对命令行参数设计解释略弱，可能较多沿用了 AI 初始代码。
```

## 七、可解释性

报告展示总分、分项分、优势证据、风险提示和答辩有效性限制。逐题记录、工具调用、隐藏验收明细和 Agent 细节不在主报告中直接展示，而是放在后台证据面板，避免学生答辩主流程暴露系统内部判断链。

## 八、局限与改进

当前系统已经支持本地规则模式和 API Agent 模式，但仍不能完全替代教师判断。后续可改进方向：

- 在更强沙箱中运行隐藏验收并支持更多题型
- 对代码进行更细粒度静态分析
- 记录答辩耗时和回答修改轨迹
- 加入班级内相似度分析
- 使用大模型对答辩回答进行更精细的语义评分
